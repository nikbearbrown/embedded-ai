# Chapter 13 — TinyML Toolchains and Deployment Pipelines

The model trains. You quantize it to int8. You prune 25% of its channels. Validation accuracy lands at 90%. You run the TensorFlow Lite converter. It prints `Conversion successful` and writes a `.tflite` file. You include the file as a byte array in your firmware, link in TFLite Micro, compile, flash to the STM32H7, run an inference — and the output is garbage. Predictions that don't match the validation set. Confidence numbers in the wrong places. A class distribution that has nothing to do with what the network learned.

You spend three hours bisecting the pipeline before you find it: the converter applied dynamic-range quantization to a layer you thought was already int8 from your QAT step. Two quantizations stacked, each with its own scale, the second computing offsets relative to a tensor that was no longer in the range it expected. The model was correct in Python and broken on the device, and *the toolchain told you the conversion succeeded the entire time*.

This is what makes deployment toolchains different from training frameworks. Training frameworks are noisy when they fail; you get a stack trace and a Python exception and a clear line number. Deployment toolchains fail silently. The converter completes, the runtime executes, the firmware boots, and the numbers are wrong. The chain from a trained `.h5` to a working binary has six steps, and any of them can introduce errors invisible until you compare on-device outputs against the original model.

## The six steps and where they break

Train. You produce a checkpoint or saved model — `.h5` for Keras, `.pth` for PyTorch, `.pb` or `SavedModel` for TensorFlow, `.onnx` for the framework-agnostic export. This artifact runs on Python with the framework runtime and a GPU. Nothing about it is yet adapted to a microcontroller.

Convert. The training model is rewritten in a deployment format — `.tflite` (a FlatBuffers serialization), `.onnx`, or a vendor-specific binary. Conversion strips training-only operations like dropout, folds batch normalization into adjacent convolutions, removes unused nodes, and prepares the graph for the runtime that will execute it.

Optimize. Quantization, structured pruning, layer fusion. Sometimes baked into the converter (TFLite's `Optimize.DEFAULT`); sometimes a separate tool (Vela for Ethos-U, TensorRT for NVIDIA, STM32Cube.AI for STM32). The output is a deployment model that has been reduced for the target.

Compile. The deployment model gets linked with an inference engine and your application code, and a binary is produced. For a microcontroller this is GCC or ARM Compiler producing an ELF. The model itself is usually embedded as a constant byte array in flash, not loaded at runtime.

Deploy. The binary is flashed via JTAG, USB, or a serial bootloader.

Verify. You run inference on the device with inputs you've also run through the original training model in Python, and you compare the outputs. If they match within a quantization-error margin, the deployment is real. If they don't, you bisect the pipeline to find which step lied.

Every step has its own failure mode. The converter can refuse a custom layer, or accept it and silently substitute an approximation. The optimizer can apply a fusion that's incompatible with the target runtime's kernel. The compiler can pack the model in a way that the runtime cannot align. The runtime can have an old bug for an op that worked everywhere else.

## TensorFlow Lite for Microcontrollers — the dominant path

TFLite Micro is the inference engine most embedded ML projects deploy through. It's a C++ library with no standard-library dependencies, no dynamic memory allocation by default, and a code footprint around 100 KB before you add op kernels. Three things make it work on a microcontroller: it's deterministic, it's static, and it's small.

The conversion call from a Keras model to a quantized `.tflite` is a few lines:

```python
import tensorflow as tf

model = tf.keras.models.load_model('my_model.h5')
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.representative_dataset = representative_dataset_gen
tflite_model = converter.convert()

with open('my_model.tflite', 'wb') as f:
    f.write(tflite_model)
```

The `representative_dataset_gen` is the calibration set — 100–1,000 inputs the converter runs through the network to record activation ranges and pick scale and zero_point per tensor. Skip this step or use a non-representative set and the resulting quantization will be wrong in ways that don't surface until on-device inference.

The model file itself is binary; you embed it in firmware as a `const unsigned char` array (typically generated by `xxd -i my_model.tflite`). Then you allocate a *tensor arena* — a static buffer that the interpreter uses for all activation memory — register the operations the model uses with an `OpResolver`, and construct an interpreter:

```cpp
constexpr int kTensorArenaSize = 200 * 1024;
alignas(16) uint8_t tensor_arena[kTensorArenaSize];

const tflite::Model* model = tflite::GetModel(my_model_data);

tflite::MicroMutableOpResolver<10> resolver;
resolver.AddConv2D();
resolver.AddDepthwiseConv2D();
resolver.AddFullyConnected();
resolver.AddSoftmax();
// ... every op the model uses must be added explicitly

tflite::MicroInterpreter interpreter(model, resolver, tensor_arena, kTensorArenaSize);
interpreter.AllocateTensors();
```

Inference is then `Invoke()` after copying input data into the input tensor and reading from the output tensor afterward. The output is int8; you dequantize using the scale and zero_point the runtime exposes through `output->params`.

Four pitfalls swallow most first-time deployments. *Tensor arena too small* — `AllocateTensors()` returns `kTfLiteError` and the message tells you exactly how many bytes you needed; bump `kTensorArenaSize` and recompile. *Missing operations* — the model uses an op you didn't add to the resolver, and you get `Op not found` at startup; add it. *Quantization mismatch* — the calibration set used at conversion didn't match what the device sees, and predictions are wrong; recalibrate with a representative set. *Memory alignment* — the tensor arena needs 16-byte alignment, which `alignas(16)` gives you; without it you get hard faults that look like random crashes.

## Edge Impulse and where the abstraction stops

Edge Impulse is the other path. It's a web platform that runs the entire pipeline behind a GUI: data collection, model design, training, optimization, and deployment-package generation. You upload data or stream it from the device. You configure preprocessing — MFCCs and FFTs for audio, scaling and normalization for images. You pick an architecture from a menu or supply a custom one. The platform trains in the cloud, quantizes automatically, and hands back a C++ library, an Arduino library, or a firmware binary tailored to your target.

What it handles well is the boilerplate. You don't write OpResolver registrations, you don't manage tensor arena sizing by hand, you don't generate `.h` files from `.tflite` files. The kernels it generates are vendor-tuned for ARM Cortex-M, Cortex-A, ESP32, and a handful of accelerators. Preprocessing is compiled into the deployment library, so the audio path on the device matches the audio path in training.

What it hides is exactly the visibility you need when something fails. You can pick architectures from a menu but you cannot drop in arbitrary custom layers. Hyperparameter control is limited compared to writing your own training script. And when the deployed model misbehaves, you don't have direct access to the converter logs, the intermediate `.tflite` file, or the optimization decisions the platform made on your behalf.

The right call is roughly: rapid prototype, team without ML expertise, standard task on a supported target — Edge Impulse, weeks become hours. Custom architecture, fine-grained quantization control, or a deployment failure that needs intermediate artifacts to debug — own the pipeline, write the converter calls yourself, and keep the artifacts.

## ONNX and vendor compilers

ONNX is the framework-agnostic intermediate format. Train in PyTorch, export to ONNX, deploy with TFLite, ONNX Runtime, or any vendor tool that consumes ONNX. The export is a single call:

```python
torch.onnx.export(model, dummy_input, 'my_model.onnx',
                  input_names=['input'], output_names=['output'])
```

The `dummy_input` matters: ONNX export traces the model, so the shape you supply is the shape baked into the graph unless you declare dynamic axes.

Most embedded AI accelerators — and many microcontrollers with NPUs — require a vendor compiler that consumes ONNX or TFLite and emits a binary tuned for the silicon. STM32Cube.AI consumes TFLite, ONNX, or Keras and emits optimized C for STM32 parts with or without NPUs; it generates a report listing supported and unsupported layers, and it will fail loudly on ops it doesn't recognize. Vela compiles TFLite for Ethos-U NPUs and demands every operation be int8 — float32 layers cause compilation to fail. The Edge TPU compiler accelerates only certain operations; everything else falls back to the host CPU, which sometimes makes a Coral-targeted model slower than a CPU-only deployment because the dispatch overhead dominates the savings.

The discipline here is to read the compilation report. Every vendor compiler tells you which ops were accelerated and which fell back. If half your model fell back to the CPU, the accelerator is doing nothing for you and you need to either redesign the model around supported ops or pick a different target.

## Toolchain-induced accuracy loss, and how to bisect it

Toolchain-induced accuracy loss is what happens when the deployment pipeline introduces errors that didn't exist in the trained model. The model was 94% in Python, the device shows 87%, and nothing about the model itself changed. Four sources cover most of the cases.

*Calibration set unrepresentative.* The converter sampled activations on data that didn't span the deployment distribution — bright images only, when the device sees mixed lighting. Quantization ranges land in the wrong place. Fix: assemble a calibration set that spans the deployment distribution, including edge cases.

*Layer fusion side effects.* Converters fuse Conv + BatchNorm + ReLU into a single kernel for performance. If the fusion has a numerical instability — and TFLite has shipped subtle ones — accuracy drops. Fix: temporarily disable fusion and check whether accuracy returns; if it does, file an issue and either pin to an older converter version or work around the affected layer.

*Operator mismatch.* PyTorch's `F.gelu()` might map to TFLite's approximate GELU rather than the exact one. The math is close, the model behavior at int8 is sometimes not. Fix: check the converter's documentation for op mappings; prefer standard ops where possible.

*Float-precision differences.* Even without quantization, float32 on a CPU differs from float32 on a GPU due to rounding-mode and SIMD-order differences. The deltas are usually under 0.01% accuracy and don't matter; if they're over 1% something larger is wrong.

The way you measure all of this is to compare the converted model's accuracy against the original on the *same* validation set, then bisect. Run the validation set through the trained model in Python and record outputs. Run it through the converted model in the TFLite interpreter (still in Python). Run it on the device. The differences between (training, converted) and (converted, on-device) localize the error to either conversion-and-optimization or compilation-and-runtime. From there, you keep bisecting — disable fusion, swap calibration sets, check op support — until the source is found.

## A keyword spotter ends up on a Nano 33 BLE Sense

A 1D CNN for keyword spotting. Five convolutional layers, two fully connected, 16 kHz audio in one-second windows, twelve output classes (ten keywords plus *unknown* and *silence*). Validation accuracy 94.2% in float32. Target an Arduino Nano 33 BLE Sense — nRF52840, Cortex-M4 at 64 MHz, 256 KB SRAM.

Convert with full int8 quantization, including int8 input and output tensors so no host-side casting is needed:

```python
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.representative_dataset = representative_dataset_gen
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.int8
converter.inference_output_type = tf.int8
tflite_model = converter.convert()
```

The converted model is 85 KB. Test it in the TFLite Python interpreter against the validation set, manually quantizing each input by `(audio / input_scale + input_zero_point).astype(np.int8)` and dequantizing the output by `(output - zero_point) * scale`. The interpreter reports 93.1% — a 1.1% loss against the float32 baseline, well inside the budget for int8 PTQ.

Convert the binary to a C array with `xxd -i keyword_model.tflite > keyword_model_data.h`. Build the Arduino sketch with an 80 KB tensor arena, register only the four ops the model actually uses (Conv2D, FullyConnected, Reshape, Softmax — every op you don't add saves a few hundred bytes of code), and wire the microphone path to copy audio into the input tensor before each `Invoke()`.

Compile, flash, run. Measured latency 42 ms against a 100 ms budget. SRAM usage: 80 KB tensor arena plus 20 KB of stack equals 100 KB on a 256 KB part. On-device accuracy on 50 recorded samples played through the microphone: 92.8%. The 0.3% gap between the TFLite interpreter (93.1%) and the device (92.8%) tracks to microphone noise and the small differences between training-time audio preprocessing and the device's audio frontend, not to the toolchain. That gap is acceptable; if it had been three percent, the next move would be to instrument the audio frontend and verify that what the device passes to the model matches what the training pipeline passed to the model.

The deployment is real. Not because the converter said `Conversion successful` — it always says that — but because the on-device numbers track the Python numbers. That comparison, run against a held-out set of inputs, is the only thing that distinguishes a working deployment from a silent failure.

## What compression and tooling cannot do

A toolchain can deploy a working model. It cannot deploy a broken one. If the trained model is below threshold before deployment, the converter will not save it. If the architecture uses operations the runtime doesn't support — attention layers on a runtime without matrix-multiply units, dynamic control flow on a static-graph engine — no compiler flag will compile the dependency away. If a model needs ten times more compute than the target offers, no quantization or fusion will close the gap.

Toolchains exist to preserve a working model across the boundary from training framework to embedded firmware. They are not optimization layers in disguise. The model that ships is whatever you fed the converter, give or take a few percentage points. The verification step is the only thing that tells you which percentage points moved.

What would change my mind: if a deployed model showed *no* accuracy drift between the TFLite interpreter and the device across many architectures and many targets — that would suggest the silent-failure mode I'm worried about is actually an artifact of a few buggy converters, not a structural property of the toolchain, and the pipeline is more trustworthy than this chapter treats it as.

Still puzzling: when on-device accuracy lands a percentage point or two below the TFLite interpreter and the math says it shouldn't, how to tell whether the gap is preprocessing drift, microphone hardware, runtime kernel differences, or a memory-alignment fault that's corrupting one weight per inference. The bisection works in principle and is exhausting in practice.
